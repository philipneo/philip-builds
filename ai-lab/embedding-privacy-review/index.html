The privacy work ships before the model does

Embedding Privacy Review.

The next Secure RAG step is a real embedding model — and the safest way to add one is to decide the privacy architecture first. This page is that review: the measured retrieval gaps that motivate embeddings, the threat model for sending anything to a provider, the architecture options, and the explicit decisions that gate Build 07.

Honest scope: nothing on this site computes embeddings today. No visitor query leaves your browser on any Neo Labs page. Keyword retrieval over the generated public corpus remains the working engine, and every number below is either measured live on this page or carries the date it was verified.

Evidence, not appetite

First: prove keyword retrieval actually falls short.

An embedding model is a privacy liability the moment it touches a query, so it has to earn its place. This probe runs 20 locked paraphrase queries — questions a real visitor might type, phrased without the page's own keywords — against the same engine the Secure RAG demo uses, in your browser, right now.

Running the paraphrase probe…

Method: a query counts as answered when any chunk a human judged as answering it appears in the engine's top 5 (hit@5). The query set and acceptable chunks were locked on 2026-07-15 after being measured offline; this page re-measures them on every load, so the number above is reproduced — never typed in. This is a targeted gap probe, not a semantic benchmark: it exists to show which phrasings keyword scoring misses, which is exactly the gap embeddings are meant to close.

Exact input set

What would — and would not — be embedded.

Embedded (build time, once per release)

  • The title and text of each generated corpus chunk — currently … chunks from … approved public pages
  • All of it is already public by construction: the corpus allowlist is the build script's only input, and you can read every chunk in the Corpus Explorer

Never embedded

  • Anything under private/ — excluded from the corpus by construction, so it cannot reach an embedding pipeline that only reads the corpus
  • Visitor queries, in the recommended default — and in any hosted mode, only with the explicit, disclosed decision in the register below
  • Queries the guardrail refuses: the guardrail-before-network invariant means a blocked query must be refused before any network call could happen

Integrity contract for the vector artifact

If vectors ship, they ship like the corpus does: a generated, committed artifact — keyed by chunk_id + content_sha1 + model + dimensions + generation date. A build_vectors.py --check mode would validate vector–corpus alignment without any API call, mirroring the corpus drift check, and the eval suite would gain stale-vector, orphan, dimension, and count checks before anything goes live.

Three ways to add embeddings

Local, hosted, or hybrid — the privacy shape differs.

Considered

Fully local (in-browser model)

An open model (e.g. a MiniLM-class encoder, ~23 MB) embeds both chunks and queries in the browser. Zero egress, total privacy.

Rejected for web delivery: a multi-megabyte dependency on a static portfolio page fails the performance budget. Viable for offline tooling.

Considered

Fully hosted

Chunks and every visitor query go to a provider API at request time.

Rejected as the default: it makes every visitor's search a third-party data event, and the demo must not do that silently.

Recommended

Hybrid, build-time first

Chunk vectors are computed once at build time (public text only) and committed as a public artifact. Query embedding happens only in an explicitly disclosed hosted mode via a same-origin /api/embed route — with the keyword engine as the always-on fallback and comparison baseline.

Best privacy-to-capability ratio, and it keeps the working system working.

Threat model

What could go wrong, and what answers it.

Visitor-query egress

Risk: hosted query embedding turns every search into a third-party data event, invisibly.

Answer: local keyword mode stays the default; hosted mode requires the explicit decision below plus an on-page disclosure the visitor can see before typing.

Guardrail bypass via the network path

Risk: an extraction-intent query gets embedded (and thus egressed) before the refusal logic sees it.

Answer: the guardrail-before-network invariant, enforced by tests: classifyQuery runs first, and a blocked query must produce zero network activity.

Vector inversion

Risk: published vectors could be partially inverted back into text.

Answer: only text that is already public — and browsable in the explorer — would ever be embedded, so inversion can reveal nothing that isn't already on these pages.

Provider retention & training

Risk: embedded text or queries are retained by the provider, or used to train models.

Answer: the provider criteria below make no-training defaults, retention windows, and stricter options an explicit selection requirement — re-verified at implementation time, not assumed.

Key exposure

Risk: an embedding API key reaches the browser.

Answer: same discipline as the existing assistant backend: keys live in server-side environment variables behind a same-origin route; this page and the artifact contain none. The secret scan runs before every release.

Stale or orphaned vectors

Risk: page copy changes, the corpus regenerates, and vectors silently describe text that no longer exists.

Answer: content-hash keying plus a no-network --check drift gate, and eval-suite contract checks that fail on any mismatch.

The gate itself

Five decisions, then Build 07 can exist.

These are deliberately not made by the review. They are the project owner's calls, recorded here so the gate is public and the eventual implementation can be checked against it.

  1. Proceed with embeddings at all?

    Recommendation: yes — the measured gap above is real, and the hybrid build-time architecture keeps privacy intact by default.

  2. Accept visitor-query egress in a hosted mode?

    Recommendation: only with an on-page disclosure visible before the first query, and with local keyword mode remaining the default experience.

  3. Which provider, and what spend cap?

    Requires confirming what is actually configured in the deployment environment and setting a hard spend limit before any key is used.

  4. Publish the vector artifact publicly?

    Recommendation: yes — it would contain only transformations of already-public text — but it is an explicit publication decision, not a default.

  5. Zero-data-retention / regional processing?

    Recommendation: request stricter options where the provider offers them; document whichever level is actually granted.

Until these are answered, nothing activates. No provider is called, no key is configured by this review, no vector is computed, and this page stays exactly as private as the rest of the site: fully static, fully local.

After the gate

What Build 07 would ship.

  • scripts/build_vectors.py — stdlib-only, key from environment, emits the committed vector artifact, with a no-network --check drift gate
  • A same-origin /api/embed route with the same server-side-key discipline as the existing assistant backend
  • Hybrid ranking that fuses keyword and vector signals, displayed with both signals visible — so the comparison against the keyword baseline stays inspectable
  • Eval-suite extensions: vector–corpus contract checks, and this page's paraphrase probe re-run against hybrid retrieval to measure the actual improvement
  • On-page mode disclosure wherever a query could leave the browser
