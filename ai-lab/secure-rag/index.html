Security & trust layer

Guardrails you can see, not just claim.

The interesting part of RAG for real businesses is not the search — it is what the system refuses to do. This prototype makes those refusals visible.

Enforced in this prototype
  • No secrets in the corpus — the index is built only from public page content, so there is nothing sensitive to leak.
  • No private files indexed — internal operating notes under private/ are excluded by construction, not by filter.
  • Extraction-intent guardrail — queries that ask to reveal keys, secrets, credentials, or private data are refused with a clear reason.
  • Route whitelist — result links can only point to known internal pages of this site.
  • Safe error states — offline, too-broad, and no-result states never invent an answer.
  • Session-only logging — the audit preview lives in memory and clears on reload; nothing is sent anywhere.
Planned controls — not built yet
  • Access control on a real retrieval API, so private corpora could ever be considered safely.
  • Persistent audit logs with a review workflow instead of an in-memory preview.
  • Embedding privacy review before any real content is embedded into a vector index.
  • Prompt regression tests for the future answer-synthesis layer.
  • Retrieval eval set so relevance is measured, not eyeballed.

Technical architecture

This is the engineering map behind the demo: what runs locally today, what is deliberately constrained, and what would need real backend infrastructure before the word "production" belonged anywhere near it.

Tokenizes the query, scores a curated public-content corpus, thresholds weak matches, and ranks the top sources. Future work: embeddings and semantic search.

Access-control layer

The safest access-control choice is zero private corpus access: only public pages are indexed, and result routes are whitelisted before render.

Blocks secret-extraction and private-data intent, detects too-broad queries, and refuses unsafe asks with a reason instead of returning fuzzy matches.

A live in-browser eval harness regression-tests refusals, hit@k rankings, honesty states, and the route whitelist. Semantic eval sets wait for real embeddings.

Threat model

Risks the demo is designed around.

The threat model is intentionally visible because secure AI work is not just "add a model." It is deciding what the system should never retrieve, never claim, never log carelessly, and never link to.

Prompt injection risk

A query tries to override the system: ignore labels, reveal secrets, or trust unverified text.

Mitigation: the prototype has no generative answer layer yet, blocks extraction intent, and only renders pre-curated public summaries.

Private-file leakage risk

A user asks for internal notes, outreach files, client lists, keys, or private repo paths.

Mitigation: private files are not in the corpus at all, and private-data queries are refused before retrieval.

Overbroad retrieval risk

A vague query like "AI" could match everything and create a misleading answer.

Mitigation: broad terms trigger a sharpen-the-question state instead of a weak result pile.

Hallucinated proof risk

The system could imply credentials, clients, benchmarks, or production infrastructure that do not exist.

Mitigation: each result carries a Live / Prototype / Planned / Learning label and a claim-safety note before render.

Logging / privacy risk

Search logs can accidentally become sensitive data if they are stored without a plan.

Mitigation: this demo logs only in memory and clears on reload. Persistent logs are planned only with retention and review rules.

Route injection risk

A result could point to an invented, external, or unsafe URL.

Mitigation: result links pass through a whitelist of known internal routes before they render.

Prototype vs production

What can be claimed today, and what cannot.

This panel is here to keep the portfolio honest. The demo proves product, architecture, security, and QA thinking; it does not claim a production RAG backend, private-data search, or measured semantic-retrieval quality.

Live now

Safe to show

  • Clickable Secure RAG demo route on the public site.
  • Local search, ranking, result cards, source links, and guardrail states.
  • In-browser regression eval harness for guardrails, rankings, and honesty states.
  • Site-wide secure /api/chat pattern for the separate AI assistant.
  • Release checks: mobile QA, link crawl, syntax checks, and secret scans.
Simulated

Prototype-only behavior

  • Retrieval runs against a hand-curated public-content index in browser memory.
  • Ranking is keyword scoring, not semantic embeddings.
  • Audit logs are a session-only preview, not persistent compliance logs.
  • Refusals model the intended product behavior before real data is involved.
Required for production

Future build requirements

  • Server-side ingestion pipeline with explicit corpus approval.
  • Embedding privacy review, vector database, access control, and route-level auth.
  • Answer synthesis with citations, source-boundary enforcement, and regression tests.
  • Monitoring for latency, cost, fallback rate, refusals, and retrieval quality.
Do not claim

Not true yet

  • Not a production vector database.
  • Not private document search.
  • Not a production compliance or private-data security system.
  • Not measured semantic retrieval accuracy.
