Shipped in Build 07 · b07-lex-1

First, earn every win that doesn't need a model.

Before adding an embedding model — with its privacy, cost, and drift surface — the keyword engine took every improvement that stays deterministic, local, and explainable. Three changes shipped:

1 · Weak morphological tier

Suffix-folded text matching ("keys" ~ "key", "measuring" ~ "measured") scores half a raw text match. It can surface results raw matching misses, but by design it cannot overturn the established ranking contract — the 42-check suite pins the winners.

2 · Corpus-grounded expansions

A seven-entry table maps visitor vocabulary onto tokens the corpus actually uses: quality assurance → QA, machine learning → ML, certification → cert, credential, website → site. Acronyms and lexeme variants only — meaning-level synonymy stays embedding territory, honestly.

3 · Single-token threshold

The keep-threshold of 2.4 was tuned for multi-word queries; a lone informative word ("certifications") matching chunk text scores 1.0 and could never qualify. Single-token queries now use a 1.0 threshold — broad single terms are still refused by the guardrail before scoring.

Result: hit@5 moved from 13/20 to 14/20 with the 42-check regression contract fully preserved — measured before and after via the same offline JavaScriptCore harness, and re-measured live above. The modest gain is the honest headline: most remaining misses are vocabulary gaps ("hire" vs "start a project") that no amount of lexical tuning closes without overfitting to the test set.

Provider-neutral design

The hybrid pipeline, layer by layer.

When the Build 06 gates clear, this is the shape of the system. Every layer left of "vector retrieval" already exists and keeps working if everything to the right fails or is never activated.

  1. Intake & validationLength limits, input normalization — exists today
  2. GuardrailsExtraction-intent and private-data refusals run first; a blocked query must produce zero network activity — the guardrail-before-network invariant, enforced by tests
  3. Disclosure stateIf hosted mode is ever enabled, the egress notice renders before the first keystroke reaches a provider
  4. Keyword retrievalThe current engine, including the b07-lex-1 layer — permanent baseline and fallback
  5. Vector retrieval (gated)Cosine over build-time chunk vectors; query embedding via same-origin route only in disclosed hosted mode
  6. FusionReciprocal-rank fusion of the two ranked lists (see below), deduplicated by chunk ID
  7. Provenance & labelsChunk IDs, content hashes, claim-safety labels, and both retrieval signals displayed — inspectable, not blended away
  8. Failure fallbackProvider timeout, rate-limit, or error → keyword results with a visible degraded-mode notice; never a blank page, never a fake answer
  9. Versioning & rollbackVectors keyed by chunk hash + model + dims + date; corpus drift fails the build check; rollback is one git revert
Fusion methodHow it worksFit for this corpus
Weighted score combinationBlend raw scores with tuned weightsPoor — keyword scores (0–15+) and cosine scores (0–1) live on incompatible scales; normalization adds tuning surface a 46-chunk corpus can't validate
Reciprocal-rank fusion (recommended)Score by rank position only: Σ 1/(k + rank), k=60Best fit — scale-free, no tuning beyond one constant, well-studied, and trivially explainable on a results card
Confidence-gated vector useVector search only when keyword confidence is lowAttractive for cost, but adds a threshold that hides which signal answered — worse for an inspectable demo
Candidate generation + rerankOne system retrieves, the other reordersOverkill at 46 chunks; justified at the scale where retrieval and ranking genuinely separate

Why RRF: this is a small, public, privacy-sensitive corpus where explainability beats optimality. Rank-based fusion means the result card can honestly say "keyword rank 2 · vector rank 1 → fused rank 1" with no hidden normalization to defend. The keyword baseline stays visible as a comparison column, so the semantic layer has to prove its value on the same probe this page already runs.

What must be true first

Activation gates, in dependency order.

  • The five Build 06 decisions — proceed, egress disclosure, provider + spend cap, vector publication, retention level — answered and recorded on the review page
  • Provider facts re-verified at implementation time (the review's facts are dated 2026-07-11/12 and expire by design)
  • build_vectors.py --check green: vectors aligned to the corpus by chunk ID + content hash, with no network call needed to verify
  • Eval extensions locked: stale-vector, orphan, dimension, and count contract checks, plus this probe re-run against hybrid retrieval — activation requires beating the 14/20 keyword baseline on held-out paraphrases, not just these 20
  • Guardrail-before-network test: every refusal case in the 42-check suite re-verified to produce zero network activity in hosted mode
  • Rollback rehearsed: one revert returns to keyword-only with no data loss, because vectors are a generated artifact, not state
