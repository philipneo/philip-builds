Live system · real site feature

AI Portfolio Assistant — the first Neo Labs proof.

The chat assistant in the corner of this site is the case study. It is a real deployed feature — not a client project, not an enterprise product, and not a fictional demo. It proves the first layer: useful AI UX connected to a secure server-side backend with honest fallback behavior.

Problem

A static portfolio needed a genuinely helpful assistant without exposing an API key in browser code, saving visitor messages in the frontend, or making fake claims about clients, prices, credentials, or results.

Solution

A Vercel serverless /api/chat endpoint that reads the key from server-side environment variables, with a rule-based browser fallback, backend health checks, and a validated routing layer for every link the assistant offers.

Cloud deployment layer

Vercel serverless route

Same-origin browser calls, deployment previews, and function logs make the backend testable without putting keys in GitHub Pages.

Security & trust layer

No browser-exposed keys

Environment variables stay server-side. Replies are sanitized, routes are whitelisted, and errors return safe fallback hints.

MLOps roadmap layer

Planned model discipline

Next proof targets include prompt regression tests, model evaluation, cost/latency tracking, and monitoring for fallback behavior.

Request flow

One same-origin path. The key never reaches the browser.

Engineering decisions

Security-aware design, not AI theater.

Security & integration
  • API key in server-side environment variables only — never in HTML, CSS, or JS.
  • Browser talks to a same-origin /api/chat; it never calls a provider URL directly.
  • GitHub Pages stays fallback-only — it cannot run the backend, so it degrades honestly.
  • Vercel backend powers real model mode behind the private key.
Guardrails & honesty
  • Route whitelist — assistant buttons can only link to known internal pages.
  • Prompt constraints fix prices ($150 / $300 / $50 mo), forbid invented claims, and keep demos labeled fictional.
  • Reply sanitizer strips placeholder # links and any invented external URLs.
  • The UI never claims a model answered unless a real reply actually returned.
  • Secret scans are part of the release checklist before AI changes ship.

Health check & states

It tells the truth about its own backend.

# → UI shows "billing/quota" fallback, never the key

What this proves

Frontend product UX wired to a backend AI service — safely, and explained.

  • I can connect a real product UI to a backend AI service without leaking secrets.
  • I can design graceful degradation so the feature never lies about its state.
  • I can add guardrails (route whitelist, prompt rules, sanitizer) to keep output safe.
  • I can debug a live deployment from Vercel logs and explain the tradeoffs in plain English.
  • I can separate shipped proof from planned ML/MLOps work without pretending the roadmap is already complete.
