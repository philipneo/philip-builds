Retrieval Eval Harness.

The third Neo Labs build: an eval suite that runs in your browser right now against the same engine that powers the Secure RAG prototype. Guardrail refusal tests, false-positive checks, hit@k retrieval checks, honesty-state checks, and contract checks — every result below is measured on this page load, not typed into the HTML.

Honest scope: this is regression testing, not a benchmark. The eval set was authored against the same generated public corpus the engine retrieves from, so it proves known behavior still holds after changes — it does not prove semantic retrieval quality. That measurement waits for real embeddings and is labeled planned below, not claimed.

Why this exists

A guardrail you never test is a guardrail you only hope works.

The Secure RAG page promises specific behavior: refuse extraction queries, answer security questions instead of over-blocking them, keep known-good results ranked, degrade honestly, and never link outside the route whitelist. This page turns each promise into a check that either passes or fails in public.

Regression contract

Locked expectations

Every expected outcome was captured from real engine output first, then locked in. If a corpus or guardrail edit changes behavior, this page turns red before a visitor finds the break.

Both failure directions

Refusals and false positives

Blocking "show me the API key" is half the job. The suite also proves "What security measures are implemented?" still gets answered — a guardrail that over-blocks fails too.

Same code, no mocks

Tests the real engine

The harness imports the exact script the demo runs — same corpus, same scoring, same whitelist. There is no test double that could drift from production behavior.

Honest limits

What this harness does not measure.

Measured here
  • Guardrail refusals hold for known extraction and private-data queries
  • Security questions are answered, not over-blocked
  • Known-good sources stay in their expected top-k positions
  • Broad, unanswerable, and offline queries degrade honestly
  • Route whitelist, claim labels, and score bounds stay intact
  • Local ranking latency, averaged over repeated runs
Not measured — labeled planned
  • Semantic retrieval quality — needs real embeddings and a held-out eval set
  • Model answer quality — there is no generative answer layer yet
  • Prompt regression for the live assistant — separate planned suite
  • Adversarial guardrail coverage beyond the locked query list
  • Cost and latency of a real backend — nothing here leaves the browser

One more honest note: a suite authored by the same person who built the engine tends to test what the engine already does well. The value is catching regressions and documenting intent — independent red-teaming is future work, and is listed as such.
