# Philip Builds Studio Collaboration Guide

This repo is Philip Miranda's working folder for building Philip Builds Studio: a small service business that sells practical website fixes, tiny tools, and simple automation outcomes to local businesses.

Use this file as the shared operating agreement for Codex, Claude, and any other assistant working in the repo.

## Current Priority

Read `private/PROJECT_STATUS.md` before deciding what to do next.

The current business priority is making Philip Builds Studio look like a real public website-building business. Email/Gmail/outreach work is paused. Help Philip get to a polished site by staying focused on:

- Strong public homepage.
- Clear service packages.
- Clickable demos and proof-of-work sections.
- Mobile-clean layout.
- Simple contact/request-a-site call to action.

## Important Constraint

Do not send client, outreach, or test emails. Do not work on email/Gmail/outreach unless Philip explicitly reopens that lane.

## Public vs Private Files

Public-facing files:

- `README.md`
- `index.html`
- `styles.css`
- `script.js`
- `portfolio/`
- `demos/`

Private operating files:

- `private/PROJECT_STATUS.md`
- `private/AGENTS.md`
- `private/DETAILING_50_PROSPECT_TRACKER.md`
- `private/FIRST_CLIENT_CLOSE_KIT.md`
- `private/REPLY_HANDLING.md`
- Other files under `private/`

Keep private notes, lead trackers, client details, and internal business decisions out of public pages unless Philip explicitly asks.

## Agent Roles

Use `private/AGENTS.md` for the local role system:

- Boss decides the next highest-leverage action.
- Hunter finds and verifies leads.
- Closer handles interested replies and protects scope.
- Builder improves public assets and demos.
- Proofreader tightens copy.
- Finance tracks quotes, deposits, balances, and maintenance follow-up.

When Philip asks "what next," start as Boss, then route to the right role.

## Collaboration Rules

- Leave a brief note in `private/PROJECT_STATUS.md` when a meaningful decision, blocker, or next action changes.
- Do not overwrite another assistant's work without reading it first.
- Keep changes scoped to the requested task.
- Prefer simple static HTML/CSS/JS unless a real client need justifies adding tooling.
- Verify public pages still open directly in a browser; this repo does not require an install step.
- Keep sales promises modest: no guaranteed revenue, no medical/legal/financial advice automation, and a human in the loop for customer-facing messages.

## Handoff Format

When handing work to another assistant, include:

- What changed.
- Files touched.
- Current blocker.
- Next best action.
- Any risks or decisions Philip needs to make.

Short, practical handoffs beat long theory.
