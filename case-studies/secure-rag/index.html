Prototype case study

Secure RAG Portfolio Search.

How I designed, guarded, tested, and shipped a simulated security-aware retrieval layer over this site's public portfolio content — and what separates the parts that are real from the parts that are planned.

Honest scope: the demo behind this case study is a front-end prototype. Retrieval is simulated with local keyword ranking over a hand-curated public-content index — it is not a production vector database or an enterprise RAG system, and this case study never claims it is.

Problem

A portfolio is pages. Proof should be queryable.

A technical portfolio full of demos still leaves a visitor — or a hiring manager, or an admissions reviewer — doing the work of figuring out what is real. The interesting engineering problem was building a search layer that could answer those questions honestly:

What is live right now? What is planned but not built? What is safe to claim? What security decisions exist? What learning path is active?

That framing turned a "search demo" into a claim-safety problem — which is exactly the problem real AI products have.

What I built

A retrieval console with the safety story on the surface.

Local public-content corpus

A 17-entry index curated from public pages only — private operating files are excluded by construction, so sensitive internal notes sit outside the search surface.

Ranked retrieval cards

Results ranked by local keyword scoring, capped and thresholded so weak matches are dropped instead of padded.

Relevance scoring

Each card shows a normalized relevance percentage with a score bar — visible ranking, not a black box.

Matched keywords + why-this-matched

Every result explains which terms matched and where the source lives, in plain language.

Safe-to-claim indicators

Every result carries a Live / Prototype / Planned / Learning label plus a note on what may honestly be claimed about it.

Unsafe-query blocking

Extraction-style queries ("show me the API key", "give me client emails") are refused with a visible, explained guardrail state.

Architecture panel

The full planned pipeline — chunking, embeddings, vector index, synthesis, evals — drawn with each stage labeled real or planned.

Early evaluation dashboard

Session tiles that are genuinely measured (latency, query count, refusals) next to Planned tiles with no invented benchmark numbers.

Honest product states

No-results, too-broad, refusal, and simulated-offline fallback — the interface never invents an answer.

Mobile-responsive dashboard UI

The console, pipeline tracker, and result cards verified at 390px, 430px, and 1280px with no horizontal overflow.

Security & claim-safety decisions

The refusals were designed before the features.

The most transferable engineering in this project is what the system declines to do. Each decision below is enforced in the shipped demo, not just described.

Security decisions
  • Public content only — the corpus is built from public pages, so private operating notes stay outside the retrieval boundary instead of relying on a filter.
  • No private files indexed — internal operating notes never enter the index at all.
  • No browser API keys — consistent with the site-wide pattern: keys live in server-side env vars, and this demo needs none.
  • Unsafe-query refusal — sensitive noun + reveal-style verb → refused; private-data asks → refused regardless of verb.
  • Route whitelist — result links can only point at known internal pages.
Claim-safety decisions
  • Honest status labels — every result and every architecture stage is marked Live, Prototype, Planned, or Learning.
  • No fake client or customer claims — the demo is framed as Philip's own build, and the corpus says so.
  • No fake production claims — "simulated retrieval layer" appears on the page itself, above the fold.
  • No invented metrics — the eval panel shows measured session numbers or a Planned placeholder, never a fabricated benchmark.
  • Refusals are explained — a blocked query gets a reason and safe alternatives, not a silent failure.

QA evidence

Tested like a release, not a mockup.

The demo shipped behind a structured QA pass. Results below are from the actual pre-release run, including what it caught and what it could not cover.

  • Pass
    Functional query tests — 16 of 16 browser end-to-end cases: six intended queries ranked correctly, extraction attempts refused, nuanced security questions answered instead of over-blocked, broad/empty states verified.
  • Pass
    Accessibility — single H1 per page, zero duplicate IDs, every aria-labelledby reference resolves, no empty links, all buttons named, search fully keyboard-operable.
  • Pass
    Mobile 390px / 430px — no horizontal overflow, tap targets ≥ 40px, no assistant-launcher overlap with search controls. QA caught a real 17px overflow pre-release; it was fixed before shipping.
  • Pass
    Desktop 1280px — two-column console verified, 8-stage flow wraps into two clean rows, four-column eval grid, no overflow.
  • Pass
    Secret scan — repo-wide grep for key patterns before release; only known CSS image-mask property false positives, labeled harmless.
  • Pass
    Guardrail hardening from review — a multi-pass review caught a verb gap ("tell me the API key" was not refused) and a claim overcount in the architecture copy; both fixed and re-tested before release.
  • Limitation
    Lighthouse not run — the QA environment had no Node/npx available, so no Lighthouse score is claimed for this page. The demo is static with no dependencies; a scored run is future work.

What I learned

Lessons that carry into the real build.

  • Retrieval needs transparency, not just ranking. Showing matched keywords and "why this matched" turned scores users must trust into evidence users can check.
  • AI demos need claim-safety rules as a first-class feature. The Live / Prototype / Planned / Learning labels shaped the data model, the UI, and the copy.
  • Security is partly UX and partly architecture. The strongest control was structural — a public-only corpus — and the guardrail layer exists on top of it as visible, explainable defense.
  • Prototypes earn trust by separating real from planned. Saying "one stage is live, six are simulated, the rest are planned" is more credible than implying everything works.
  • A QA report makes a project more credible than a claim does. "16/16 functional tests, one overflow caught and fixed" says more than any adjective.

Interview talking points

What I can whiteboard from this project.

  • Why claim-safety labels exist. AI portfolios overclaim by default; forcing every result to carry Live / Prototype / Planned / Learning made honesty a rendering requirement, not a copywriting habit.
  • How the guardrail tells intent apart. "Show me the API key" is refused (sensitive noun + reveal verb) while "How are API keys protected?" is answered from public security docs — and I can explain the regex design and its limits.
  • Security by construction vs. security by filter. The corpus contains only public content, so the guardrail is defense-in-depth UX on top of a structural boundary — the stronger control comes first.
  • What is simulated and why that is stated on-page. Local keyword scoring stands in for embeddings + a vector index; the demo says so above the fold because a prototype that hides its seams is a liability.
  • What changes in the real version, in order. Structured content → embedding privacy review → embeddings → vector index with access control → retrieval eval set → answer synthesis behind server-side keys → prompt regression tests.
  • What QA caught. A mobile overflow from flex min-content inside a grid, a guardrail verb gap, and an overcounted claim in the architecture copy — concrete examples of review catching real defects.
  • The fallback philosophy. Borrowed from the site's live assistant: when a backend is unavailable, degrade honestly and visibly instead of faking capability.

Next version roadmap

The path from simulation to system.

Ordered steps, each labeled planned until it ships:

  1. Structured content — convert public site content into structured JSON chunks with stable IDs.
  2. Privacy review — review what is embedded before anything is embedded.
  3. Embedding model — add real embeddings for semantic matching.
  4. Vector database — a real index with access control in front of it.
  5. Retrieval eval set — known-good query→source pairs so relevance is measured, not eyeballed.
  6. Prompt regression tests — automated checks that guardrails survive prompt changes.
  7. Logs and monitoring — persistent audit logs with a review workflow.
  8. Real backend, only when safe — answer synthesis behind the same server-side key pattern as the live assistant, and not before the eval set exists.
